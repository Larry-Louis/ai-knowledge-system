# Docs Governance

本文档用于约束本仓库技术文档的维护策略，减少重构后的信息漂移。

## 目标

- 保证“架构描述”和“代码实际行为”一致。
- 降低文档维护成本，避免频繁修正行号。
- 明确哪些文档是规范，哪些文档是历史记录。

## 写作与引用规则

- 优先使用“模块路径 + 函数名”描述实现位置。
- 除非在做一次性审计报告，避免固化精确行号。
- 描述流程时以阶段职责为主，不以局部实现细节为主。

## 文档分层

- `docs/architecture/*`: 当前有效的架构与约束文档（应持续维护）。
- `docs/specs/*`: 规格与实现说明文档（允许包含历史信息，但需标注时效性）。
- `docs/roadmap/*`: 里程碑与计划（可包含时间快照，不作为代码真值）。

## 变更触发器

以下改动发生时，必须同步更新文档：

- 新增/替换 application 层门面（例如 `prompt_builder`、`llm_gateway`）。
- 关键流水线入口变更（`memory_service`、`memory_pipeline_service`）。
- 分层依赖规则变化。

## 删除策略

- 不可维护、与当前代码语义冲突、且不再被索引引用的文档应删除。
- 删除前确认无外部文档链接依赖（至少全仓搜索一次）。
